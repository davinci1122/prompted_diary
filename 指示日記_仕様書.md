# 指示日記 ─ 仕様書・開発プラン v1.2

## 1. コンセプト

毎日0時、その日だけの「指示」が解禁される。指示はすべて「日常を異なる視点で見る」ためのお題であり、ユーザーは指示に従って写真を1枚撮り、カレンダーに登録する。1年が終わると、366の視点で切り取られた自分の日常のアーカイブ（＝日記）が完成する。

## 2. 確定済みの前提

| 項目 | 決定内容 |
|---|---|
| 保存方式 | クラウド（Supabase）が正。端末内（IndexedDB）はキャッシュ。キャッシュを消してもログインで復元 |
| 指示の割当 | 毎年ランダムに入れ替え（年ごとに全指示をシャッフル） |
| 登録可能期間 | 当日 0:00〜23:59 のみ。逃した日は空白のまま確定 |
| プラットフォーム | ブラウザアプリ（PWA、スマホ優先のレスポンシブ） |

## 3. 機能要件

### 3.1 カレンダー（メイン画面）
- 月表示。**月曜始まりがデフォルト**（設定で日曜始まりに切替可）
- 各セルには**サムネイル写真のみ**を表示。指示はセルをタップした日詳細で見せる
- 当日セルをハイライト。未登録の当日は「撮影ボタン」を表示
- 未来日はロック表示（指示は見えない）
- 過去の未登録日は空白セル（グレー）。タップすればその日の指示は閲覧可能
- 月送り／年送り、今日へジャンプ
- アプリを開いた時点で当日が未登録なら、控えめなバナーで知らせる（Phase 3のプッシュ通知までの代替。単純な画面内表示のみとし、不具合リスクを最小化する）

### 3.2 指示の解禁
- 端末時刻で 0:00 になった瞬間、当日の指示が閲覧可能になる
- アプリ起動中に日付をまたいだ場合も自動で切り替え（1分毎の日付チェック）
- 未来の指示はUI上完全非表示（※ローカルアプリのため技術的秘匿は不可能。仕様として「見ないことを前提としたUI」とする）

### 3.3 撮影・登録
- `<input type="file" accept="image/*" capture="environment">` でカメラ起動、またはギャラリーから選択
- プレビュー → 任意の短いメモ（140字以内）→ 登録
- 1日1枚。当日中のみ差し替え・削除可。23:59を過ぎたら編集不可
- 登録時にクライアント側で圧縮（本体：長辺1600px / WebP品質0.8、サムネ：長辺240px）

### 3.4 日詳細ビュー
- セルタップで開く。写真（拡大表示）、指示全文、メモ、登録日時
- 前日／翌日へスワイプ移動

### 3.5 その他
- ギャラリービュー（写真だけをグリッドで一覧、年間振り返り用）
- 連続記録日数・年間達成率の表示
- エクスポート／インポート（写真＋メタデータをzipで書き出し・復元。バックアップ兼クラウド移行の布石）

## 4. 指示割当ロジック（毎年ランダム）

決定論的シャッフルを採用する。乱数シード＝西暦年とし、`mulberry32(year)` ＋ Fisher–Yates で366個の指示配列をシャッフルし、day-of-year（1〜366、2/29含む通し番号）で引く。

- 同じ年なら何度計算しても同じ割当 → 再インストール・機種変更・将来のクラウド同期後も割当を完全再現できる（割当自体は保存不要）
- 平年は2/29に相当するインデックスを1つ飛ばすだけ（その年は365個を使用、1個が休眠）
- 登録済みエントリには使用した `promptId` を記録し、後からシャッフル実装を変えても過去の表示が壊れないようにする

## 5. データ設計

指示366個は静的アセット（`prompts.json`）としてアプリに同梱。`category` は内部管理・開発用であり、**UIには一切表示しない**。ユーザーデータは IndexedDB（Dexie.js）に保存。

```
prompts.json : { id: 1..366, text: string, category: string }

IndexedDB
├ entries  : { date: "YYYY-MM-DD"(PK), promptId, photo: Blob, thumb: Blob,
│              note?: string, createdAt, updatedAt }
└ settings : { key(PK), value }   // テーマ、週の開始曜日 など
```

- 容量見積り：1枚 約250〜400KB × 366日 ≒ 100〜150MB/年 → 起動時に `navigator.storage.persist()` を要求し、使用量を設定画面に表示
- iOS Safari のストレージ退避（eviction）対策として、PWAインストールと定期エクスポートを促す

## 6. 技術スタック

| 層 | 採用技術 | 理由 |
|---|---|---|
| フロント | React + TypeScript + Vite | 一般的で保守しやすい |
| 状態/DB | Dexie.js (IndexedDB) | Blob保存とクエリが容易 |
| PWA | vite-plugin-pwa (Service Worker) | オフライン動作・ホーム画面追加 |
| 画像処理 | Canvas API（リサイズ/WebP変換） | 依存ライブラリ不要 |
| バックアップ | JSZip | zipエクスポート/インポート |

クラウド拡張への備え：データアクセスを `DiaryRepository` インターフェース（get/put/delete/list/export）に集約し、実装を `LocalRepository` として提供。将来 `CloudRepository`（Supabase等：Auth＋Postgres＋Storage）を差し替え、`updatedAt` ベースの単純同期（同一dateは新しい方優先）を追加する。

## 7. 画面一覧

1. 月カレンダー（ホーム）
2. 日詳細
3. 撮影・登録モーダル
4. ギャラリー（年間グリッド）
5. 設定（テーマ / 週の開始 / エクスポート・インポート / ストレージ使用量）

## 8. 非機能要件

- 完全オフライン動作（初回ロード後はネット不要）
- スマホ縦画面を基準に設計、PCでも閲覧可能
- 端末時刻の巻き戻しによる「過去日登録」は仕様上防がない（個人利用のため）
- 写真は端末外に送信されないことを設定画面に明記

## 9. 開発ロードマップ

- **Phase 1（MVP）**：カレンダー表示、0時解禁、当日撮影・登録、日詳細、圧縮保存
- **Phase 2**：ギャラリー、連続記録/達成率、メモ、zipエクスポート/インポート、PWA化
- **Phase 3**：クラウド同期（アカウント）、プッシュ通知（0時のお知らせ／撮り忘れリマインド）、年間まとめ画像の自動生成

## 10. 決定事項（2026-07-04 確定・本文反映済み）

1. 週の開始：**月曜**がデフォルト（設定で切替可）
2. カレンダーセル：**写真のみ**表示。指示はタップで表示
3. 撮り忘れリマインド：アプリ起動時のバナー表示で代替（実装は最小限にし堅牢性を優先）。プッシュ通知はPhase 3
4. 指示カテゴリ：ユーザーには**非表示**（内部データとしてのみ保持）
